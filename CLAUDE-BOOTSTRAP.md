# Obsidian Vault 指南 - Bootstrap 模板

**Claude Code + Obsidian 入门**

## 快速设置

1. **每次会话开始**：运行 `git pull` 同步最新更改
2. **更改后**：提交并推送以保留你的工作
3. **使用内置工具**：优先使用 WebSearch 和 WebFetch 获取网页内容

## 版本控制最佳实践

**关键 - 每次会话开始**：始终在每个新 Claude 会话开始时运行 `git pull`，以确保你拥有远程仓库的最新更改。

**提交流程**：

- 创建新笔记后：`git add .` → `git commit -m "message"` → `git push`
- 重要编辑后：立即提交并推送
- 使用 `git status` 检查修改
- 当代理修改文件时：始终提交这些更改

## 文件夹结构（PARA 方法）

```
vault/
├── 00_Inbox/           # 临时捕捉点
├── 10_Projects/        # 有时间限制的倡议
├── 20_Areas/           # 持续责任
├── 30_Resources/       # 参考材料
├── 40_Archive/         # 已完成/非活动项目
├── 50_Attachments/     # 图像、PDF 等
│   └── Organized/      # 已处理的附件
└── 60_Metadata/        # 文档和模板
    ├── Reference/      # 指南和标准
    ├── Plans/          # 战略文档
    └── Templates/      # 可重用结构
```

## PARA 方法详情

### Projects (10)

- 具有明确完成标准的有时间限制的倡议
- 示例：撰写论文、开发演示文稿
- 建议的子文件夹：Research/、Drafts/、References/、Output/

### Areas (20)

- 没有结束日期的持续责任
- 示例：健康、财务、职业发展
- 创建包含相关资源链接的专用笔记

### Resources (30)

- 感兴趣的主题供参考
- 按主题组织的知识库
- 用于不与特定项目绑定的信息

### Archive (40)

- 已完成或非活动的项目
- 保持与活动部分相同的文件夹结构
- 定期审查以重新激活

## 收件箱管理

### 核心原则

- 收件箱是临时的，不是永久存储
- 使用 捕捉 → 处理 → 组织 工作流每周处理
- 随时保持 <20 个项目

### 保留在收件箱中的文件

- **关键**：带编号前缀（00、10-60）的文件永久保留
- 最近的每日/每周摘要（最近 3 个月）
- 正在处理的活跃笔记

### 处理工作流

1. 删除过时信息
2. 将相关材料移到 PARA 位置
3. 将行动转换为项目任务
4. 用 `#needs-processing` 标记需要更多处理的项目

## 文件组织指南

### 命名约定

- 每日笔记：`YYYY-MM-DD - 主题`
- 会议笔记：`会议 - [主题] - YYYY-MM-DD`
- 想法：`想法 - [简短描述]`
- 资源：`资源 - [主题] - [来源]`

### 移动规则

- 使用 `mv` 命令（不是 `cp`）以避免重复
- 首先验证目标文件夹存在
- 移动后更新内部链接
- 组织时添加 YAML frontmatter

## 附件管理

### 组织

- 将所有非文本文件存储在 `50_Attachments/` 中
- 已处理的文件 → `50_Attachments/Organized/`
- 命名：`[相关笔记]_[描述].[扩展名]`

### 辅助脚本

```bash
pnpm attachments:list        # 列出未处理的文件
pnpm attachments:organized   # 计算已组织的文件
pnpm attachments:orphans     # 查找未引用的文件
pnpm attachments:update-links # 移动后更新链接
```

## 网页内容工作流

### 内置工具（首选）

- **WebSearch**：用于一般网页搜索
- **WebFetch**：用于特定 URL
- 根据内容类型保存到适当的文件夹

### 自定义脚本（需要时）

- 单个 URL：`pnpm firecrawl:scrape <url> <output>`
- 批量 URL：`pnpm firecrawl:batch <url1> <url2>`
- 使用 frontmatter 保存到 `00_Inbox/Clippings/`

## 写作风格指南

### 结构

- 使用 `[[WikiLinks]]` 进行内部引用
- 包含 YAML frontmatter（日期、标签、状态）
- 一致的 Markdown 格式
- 具体、一致的标签

### 风格偏好

- 直接和自信的陈述
- 避免陈词滥调的过渡
- 让陈述独立存在
- 没有不必要的引导

## AI 助手指南

### 任何组织之前

1. 映射完整的文件夹结构：`find . -type d | sort`
2. 在 `60_Metadata/STRUCTURE.md` 中记录
3. 验证所有目标文件夹存在

### 处理内容

- 尊重编号的核心文件（永远不要移动 00、10-60 前缀的文件）
- 组织时始终使用 `mv` 而不是 `cp`
- 保留并更新双向链接
- 添加适当的 YAML frontmatter

### 仅使用简单命令

- **必需**：直接的、基本的命令，无需过滤
- **禁止**：复杂的正则表达式、管道命令、带过滤器的 find
- 正确示例：`ls -1` 然后手动选择文件
- 错误示例：`ls | grep pattern` 或 `find . -name "*.png"`

## 每日工作流

### 一天开始

1. 运行 `git pull`
2. 检查收件箱中的待处理项目
3. 审查活跃项目

### 一天结束

1. 处理新的收件箱项目
2. 提交并推送更改
3. 更新项目笔记

### 每周回顾

1. 处理整个收件箱
2. 归档已完成的项目
3. 更新领域笔记
4. 审查和整合资源

## 项目生命周期

### 启动项目

1. 在 `10_Projects/[项目名称]` 中创建文件夹
2. 添加子文件夹：Research/、Drafts/、Output/
3. 创建包含目标和时间线的 README

### 项目进行中

- 将所有相关材料保留在项目文件夹中
- 链接到相关资源和领域
- 定期提交以跟踪进度

### 完成项目

1. 创建项目摘要笔记
2. 将整个文件夹移到 `40_Archive/`
3. 更新相关领域笔记
4. 使用完成消息提交

## 最佳实践

### 组织

- 保持文件夹结构浅层（最多 3 层）
- 只有 7 个以上相关笔记时才创建子文件夹
- 使用链接而不是深度嵌套
- 在主要文件夹中包含 README

### 内容创建

- 先捕捉，后组织
- 每个笔记一个想法
- 大量使用链接
- 一致地标记

### 维护

- 每周收件箱处理
- 每月项目审查
- 每季度归档清理
- 定期 git 提交
